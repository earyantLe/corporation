## 技能描述

本技能用于设计系统技术架构，包括技术选型、架构设计和实施规划。

**使用者**: CTO

**适用场景**:
- 新项目技术方案设计
- 现有系统架构评审
- 技术选型评估
- 架构优化改造

**依赖技能**:
- `tech_decision` - 技术决策评估

## 使用示例

### 示例 1：电商网站架构设计

**输入**:
```markdown
🏗️ 技术架构

**项目**: 电商网站开发
**架构类型**: 新设计
**约束条件**: 5 天交付，支持 1000 并发，预算¥100,000
```

**输出**:
```markdown
🏗️ 技术方案

**项目**: 电商网站
**版本**: v1.0

## 技术选型
| 类别 | 技术 | 理由 | 备选方案 |
|------|------|------|----------|
| 前端 | React 18 + TypeScript | 生态丰富，类型安全 | Vue 3 |
| 后端 | FastAPI + Python | 开发效率高，自动文档 | Node.js + Express |
| 数据库 | PostgreSQL | 稳定可靠，支持复杂查询 | MySQL 8 |
| 缓存 | Redis | 高性能，支持多种数据结构 | Memcached |
| 部署 | Docker + Nginx | 容器化部署，易于扩展 | 传统部署 |

## 架构设计
```
用户 → Nginx(负载均衡) → 后端 API 集群 → PostgreSQL 主从
                    ↓
                Redis 缓存层
                    ↓
              对象存储 (图片/静态资源)
```

## 核心模块
1. 用户模块：注册登录、权限管理、个人中心
2. 商品模块：商品展示、搜索、分类管理
3. 订单模块：下单、支付、物流跟踪
4. 后台模块：商品管理、订单管理、数据统计

## 实施计划
| 阶段 | 内容 | 时间 | 里程碑 |
|------|------|------|--------|
| 阶段一 | 项目初始化、用户模块 | Day 1 | 用户系统完成 |
| 阶段二 | 商品模块、订单模块 | Day 2-3 | 核心功能完成 |
| 阶段三 | 支付集成、测试 | Day 4 | 支付联调完成 |
| 阶段四 | 部署上线、文档 | Day 5 | 正式上线 |

## 风险评估
| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 支付安全风险 | 中 | 高 | 使用第三方支付，减少自身存储 |
| 高并发性能 | 低 | 中 | 引入缓存和 CDN |
| 进度延期 | 中 | 高 | 优先保证核心功能，砍掉次要功能 |
```

3. 非核心模块可尝试新技术

**重新提交门下省审议**

需要同步的部门:
@CEO @门下省 @尚书省

## 检查清单

### 必查项

```markdown
- [ ] 技术选型理由充分，有备选方案
- [ ] 架构设计图清晰，数据流明确
- [ ] 实施计划可行，时间合理
- [ ] 风险评估全面，有应对措施
- [ ] 核心模块职责清晰，边界明确
```

### 选查项

```markdown
- [ ] 性能预估和压测计划
- [ ] 成本估算和预算对比
- [ ] 监控和告警方案
- [ ] 灾备和回滚方案
```

---

## 注意事项

1. 技术选型要考虑团队能力，不要盲目追新
2. 架构设计要预留扩展性，但不过度设计
3. 风险评估要诚实，不隐瞒潜在问题
4. 实施计划要留有余地，应对突发情况
5. 方案文档要简洁，控制在 2000 字以内
