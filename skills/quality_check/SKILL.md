## 技能描述

本技能用于检查各部门完成的产出质量，确保符合标准和规范。

**使用者**: QA

**适用场景**:
- 代码质量检查
- 文档完整性检查
- 设计稿检查
- 报告质量检查

---

## 使用方法

### 命令格式

```markdown
✅ 质量检查

**检查对象**: {作业/代码/文档名称}
**检查类型**: {代码审查/文档检查/设计检查}
**检查标准**: {规范名称/标准要求}
```

### 输出格式

```markdown
📋 质检报告

**检查对象**: {对象}
**检查日期**: {日期}
**检查人**: QA

**检查维度**:
| 维度 | 评分 | 说明 |
|------|------|------|
| {维度} | ⭐⭐⭐⭐ | {说明} |

**问题清单**:

**改进建议**:
1. {建议 1}

```

## 使用示例

### 示例 1：代码审查

**输入**:
```markdown
✅ 质量检查

**检查对象**: 用户认证模块
**检查类型**: 代码审查
**检查标准**: PEP8 + 安全规范
```

**输出**:
```markdown
📋 质检报告

**检查对象**: 用户认证模块
**检查日期**: 2024-01-15
**检查人**: QA

**检查维度**:
| 维度 | 评分 | 说明 |
|------|------|------|
| 代码规范 | ⭐⭐⭐⭐ | 整体符合 PEP8 |
| 代码质量 | ⭐⭐⭐⭐ | 结构清晰 |
| 安全合规 | ⭐⭐⭐⭐⭐ | 加密验证完善 |
| 测试覆盖 | ⭐⭐⭐ | 覆盖率 75% |

**问题清单**:
| 序号 | 问题 | 严重程度 | 建议 |
|------|------|----------|------|
| 1 | 函数过长 (>50 行) | 中 | 拆分为小函数 |
| 2 | 缺少边界测试 | 中 | 添加边界用例 |
| 3 | 注释不完整 | 低 | 补充 docstring |

**整体评价**: ⚠️ 需改进

**改进建议**:
1. 拆分 auth_login 函数（当前 80 行）
2. 添加密码特殊字符、空值等边界测试
3. 完善公共函数的文档字符串

**复测时间**: 修改后重新提交
```

✅ 质量检查

## 检查清单模板

### 代码检查清单

```markdown
**必查项**:
- [ ] 无语法错误
- [ ] 变量命名规范
- [ ] 函数有注释
- [ ] 有类型注解
- [ ] 测试用例完整
- [ ] 无安全隐患

**选查项**:
- [ ] 性能优化
- [ ] 错误处理完善
- [ ] 日志记录完整
```

### 文档检查清单

```markdown
**必查项**:
- [ ] 结构清晰有层次
- [ ] 有概述和说明
- [ ] 有使用示例
- [ ] 格式统一
- [ ] 术语一致

**选查项**:
- [ ] 有版本号
- [ ] 有更新日期
- [ ] 有 FAQ 章节
```

---

## 注意事项

1. 检查要对事不对人，客观公正
2. 指出问题的同时要给出改进建议
3. 严重问题必须修复才能通过
4. 轻微问题可以记录后放行
5. 检查报告要存档备查

## 打回重做流程

```
 QA 出具「❌ 未通过」报告
     │
     ▼
 CEO 审阅 → 确定责任部门
     │
     ▼
 责任部门 24 小时内修复
     │
     ▼
 提交复测申请（附修复说明）
     │
     ▼
 QA 优先复测（4 小时内完成）
     │
     ├─ 通过 ──→ 继续流程
     │
     └─ 仍不通过 ──→ 升级至 CEO 协调
```

### 复测申请格式

```markdown
🔄 复测申请

**任务 ID**: TASK-{YYYYMMDD}-{序号}
**原质检报告 ID**: QC-{日期}-{序号}

**修复内容**:
| 问题序号 | 修复方案 | 修改内容 |
|----------|----------|----------|
| 1 | 拆分函数 | auth_login 拆分为 3 个子函数 |
| 2 | 添加测试 | 新增 5 个边界测试用例 |

**自测结果**: ✅ 通过
**申请复测时间**: {时间}
```

### 复测响应 SLA

|--------|---------|---------|
| P0 | 30 分钟 | 4 小时 |
| P1 | 1 小时 | 8 小时 |
| P2 | 2 小时 | 24 小时 |
