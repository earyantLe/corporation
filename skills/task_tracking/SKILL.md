# 📋 任务追踪 · Task Tracking

> 任务状态同步和追踪机制

---

## 技能描述

本技能用于同步任务状态、追踪任务进度，支持多任务并发管理。

**使用者**: CEO, COO, QA

**适用场景**:
- 任务执行中进度同步
- 多任务并发管理
- 风险预警和阻塞上报
- 任务生命周期追溯

---

## 任务 ID 格式

```
TASK-{YYYYMMDD}-{序号}
```

示例：`TASK-20260324-001`

---

## 任务状态机

```
assigned ──→ in_progress ──→ qa_review ──→ approved ──→ completed
                │                │
                │                └──→ rejected ──→ in_progress
                │
                └──→ blocked ──→ 等待协调
```

| 状态 | 说明 | 颜色 |
|------|------|------|
| `assigned` | 已分配，等待执行 | 🟡 |
| `in_progress` | 执行中 | 🟢 |
| `blocked` | 已阻塞，需要协调 | 🔴 |
| `qa_review` | QA 检查中 | 🟠 |
| `approved` | QA 通过，等待审批 | 🟣 |
| `rejected` | QA 未通过，打回重做 | ⚫ |
| `completed` | 已完成 | ✅ |

## 使用方法

### 任务状态同步格式

```markdown
📋 任务状态同步

**任务 ID**: TASK-{YYYYMMDD}-{序号}
**任务名称**: {名称}
**当前状态**: 🟢 进行中 / 🟡 有风险 / 🔴 已阻塞 / ✅ 已完成

**主责部门**: {部门}
**协作部门**: {部门 1, 部门 2}

**分任务状态**:
| 部门 | 状态 | 进度 | 预计完成 | 阻塞原因 |
|------|------|------|----------|----------|
| 工程 | 🟢 | 60% | Day 4 | - |
| 设计 | ✅ | 100% | - | - |
| QA | 🟡 | 0% | Day 5 | 等待工程完成 |

**风险预警**:
{风险描述及应对建议}

**需要同步的部门**:
@{部门 1} @{部门 2}
```

---

### 任务分配格式（带状态追踪）

```markdown
📋 任务分配

**任务 ID**: TASK-{YYYYMMDD}-{序号}
**任务名称**: {任务名称}
**优先级**: P0/P1/P2
**状态**: assigned

**分配方案**:
| 部门 | 负责人 | 任务内容 | 截止时间 | 状态 |
|------|--------|----------|----------|------|
| {部门} | {Agent} | {内容} | {时间} | assigned |

**要求**:
1. {质量要求}
```

---

### 打回重做申请格式

```markdown
🔄 复测申请

**任务 ID**: TASK-{YYYYMMDD}-{序号}
**原质检报告 ID**: QC-{日期}-{序号}

**修复内容**:
| 问题序号 | 修复方案 | 修改内容 |
|----------|----------|----------|
| 1 | 拆分函数 | auth_login 拆分为 3 个子函数 |
| 2 | 添加测试 | 新增 5 个边界测试用例 |

**自测结果**: ✅ 通过
**申请复测时间**: {时间}
```

## 使用示例

### 示例 1：任务状态同步

**COO**:
```markdown
📋 任务状态同步

**任务 ID**: TASK-20260324-001
**任务名称**: 电商网站开发
**当前状态**: 🟢 进行中

**主责部门**: CTO
**协作部门**: 工程、设计、QA

**分任务状态**:
| 部门 | 状态 | 进度 | 预计完成 | 阻塞原因 |
|------|------|------|----------|----------|
| CTO | ✅ | 100% | Day 1 | - |
| 工程 | 🟢 | 60% | Day 4 | - |
| 设计 | ✅ | 100% | Day 3 | - |
| QA | 🟡 | 0% | Day 5 | 等待工程完成 |

**风险预警**:
无重大风险，项目按计划进行。

**需要同步的部门**:
@CEO @CTO @工程 @设计 @QA
```

### 示例 2：任务阻塞上报

**工程**:
```markdown
📋 任务状态同步

**任务 ID**: TASK-20260324-001

**当前状态**: 🔴 已阻塞

**需要同步的部门**:
@CEO @CTO @QA
```

---

### 示例 3：QA 打回重做

**QA**:
```markdown
📋 质检报告

**任务 ID**: TASK-20260324-001
**检查对象**: 用户认证模块
**整体评价**: ❌ 未通过

**严重问题**:
| 序号 | 问题 | 影响 | 修复建议 |
|------|------|------|----------|
| 1 | 密码明文存储 | 安全漏洞 | 使用 bcrypt 加密 |
| 2 | 无 SQL 注入防护 | 安全风险 | 使用参数化查询 |

**处理决定**:
- 状态变更为：rejected
- 打回工程部重新开发
- 修复后提交复测申请

**需要同步的部门**:
@CEO @CTO @工程
```

---

## 注意事项

1. **任务 ID 唯一性**: 每个任务必须有唯一 ID，格式为 `TASK-{YYYYMMDD}-{序号}`
2. **状态及时更新**: 任务状态变化时，24 小时内同步
3. **阻塞立即上报**: 发现阻塞风险，立即上报 CEO/COO 协调
4. **复测优先处理**: QA 复测申请优先处理，4 小时内完成
5. **历史可追溯**: 通过任务 ID 可追溯完整生命周期
