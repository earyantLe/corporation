## 技能描述

本技能用于规范和标准化工作状态报告，确保信息透明和高效沟通。

**使用者**: 全员

**适用场景**:
- 日报/周报
- 项目进度同步
- 阶段性成果汇报
- 临时状态更新

**依赖技能**:
- `task_tracking` - 任务追踪（数据来源）

---

## 使用方法

### 命令格式

```markdown
📊 状态报告

**报告类型**: {日报/周报/项目报告/临时报告}
**报告人**: {角色}
**报告周期**: {时间范围}
```

### 输出格式

```markdown
📊 {日报/周报}

**报告人**: {姓名}
**日期**: {YYYY-MM-DD}
**周期**: {时间范围}

## 本日/本周完成
| 任务 | 状态 | 进度 | 说明 |
|------|------|------|------|
| {任务} | ✅ | 100% | {说明} |

## 下日/周计划
| 任务 | 优先级 | 预计时间 | 说明 |
|------|--------|----------|------|
| {任务} | P0/P1/P2 | {时间} | {说明} |

## 风险和问题
| 问题 | 影响 | 需要支持 |
|------|------|----------|
| {问题} | {影响} | {支持} |

## 备注
{其他需要同步的信息}
```

### 示例 3：异常处理 - 项目延期报告

**输入**:
```markdown
📊 状态报告

**报告类型**: 临时报告
**报告人**: Engineering
**问题**: 项目可能延期
```

**输出**:
```markdown
🚨 项目风险报告

**报告人**: Engineering
**日期**: 2026-03-28
**项目**: 电商网站开发

## 风险概述
**风险等级**: 🟡 中等
**可能延期**: 2-3 天
**影响范围**: 订单模块、支付模块

## 原因分析
1. **技术难点**: 支付接口集成复杂度超预期
2. **需求变更**: 新增会员系统需求
3. **人力因素**: 1 名工程师病假

## 应对方案
### 方案 A：加班赶工
- 优点：不砍功能
- 缺点：团队疲劳，质量风险
- 建议：不推荐

### 方案 B：砍掉次要功能
- 优点：保证核心功能按时上线
- 缺点：用户体验不完整
- 建议：推荐

### 方案 C：延期上线
- 优点：功能完整，质量保证
- 缺点：影响市场计划
- 建议：备选

## 建议
采用方案 B，优先保证核心功能（用户、商品、订单）上线，会员系统延后到 v1.1

**需要决策**:
@CEO @CTO 确认方案

**需要同步的部门**:
@CEO @CTO @COO
```

## 检查清单

### 必查项

```markdown
- [ ] 数据准确，不夸大不缩小
- [ ] 进度真实，有依据
- [ ] 风险及时上报，不隐瞒
- [ ] 计划可行，不拍脑袋
- [ ] 问题清晰，有建议方案
```

### 选查项

```markdown
- [ ] 指标对比（环比/同比）
- [ ] 图示化（燃尽图等）
- [ ] 客户反馈
```

---

## 注意事项

1. 报告要简洁，不说空话套话
2. 数据要准确，有来源依据
3. 风险要及时上报，不捂盖子
4. 问题要有解决方案，不只提问题
5. 报告要定期发送，形成习惯
