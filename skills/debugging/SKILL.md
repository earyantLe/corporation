## 技能描述

本技能用于系统性调试和排查技术问题，包括日志分析、问题复现、根因定位和修复。

**使用者**: Engineering

**适用场景**:
- 线上 Bug 排查
- 测试失败调试
- 性能问题分析
- 异常错误定位

**依赖技能**:
- `coding` - 代码开发（修复问题）
- `unit_testing` - 单元测试（验证修复）

## 使用示例

### 示例 1：线上 Bug 排查

**输入**:
```markdown
🐛 调试排查

**问题**: 用户反馈登录失败，提示"服务器错误"
**环境**: 生产环境
**影响**: 所有用户无法登录
**优先级**: P0
```

**输出**:
```markdown
📋 问题排查报告

**问题 ID**: BUG-20260328-001
**问题**: 用户登录失败，提示"服务器错误"
**优先级**: P0
**状态**: ✅ 已解决

## 问题现象
- 用户点击登录按钮后，前端显示"服务器错误"
- 后端日志显示数据库连接超时
- 影响所有用户无法登录

## 影响范围
- 影响模块：用户认证
- 影响用户：100% 用户
- 影响程度：严重（核心功能不可用）

## 排查过程
1. **查看应用日志**: 发现大量数据库连接超时错误
2. **检查数据库状态**: 数据库 CPU 100%，连接数已满
3. **检查慢查询**: 发现一条全表扫描的慢查询占用大量资源
4. **定位代码**: 问题代码位于 report_service.py 第 45 行

## 根因分析
**直接原因**: 慢查询导致数据库连接池耗尽

**根本原因**:
- 报表查询缺少索引
- 没有查询超时限制
- 高峰期执行大表全表扫描

## 修复方案
**临时修复**（已执行）:
```sql
-- 终止慢查询
SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state='active';
```

**永久修复**:
```python
# 添加查询超时限制
@app.get("/api/reports")
@timeout(5.0)  # 5 秒超时
def get_reports():
    # 添加索引提示
    return db.query(Report).with_hint(Report, "INDEX(report_user_idx)").all()
```

## 验证结果
- [x] 本地复现通过
- [x] 测试环境验证通过
- [x] 生产环境验证通过

## 预防措施
1. 添加慢查询监控告警（>1 秒）
2. 所有查询添加超时限制
3. 代码审查必须包含索引检查
4. 定期分析和优化慢查询

**需要同步的部门**:
@CTO @CEO @QA
```

### 示例 2：测试失败调试

**输入**:
```markdown
🐛 调试排查

**问题**: 单元测试随机失败，错误率约 10%
**环境**: CI/CD 流水线
**影响**: 无法合并代码
**优先级**: P1
```

**输出**:
```markdown
📋 问题排查报告

**问题 ID**: BUG-20260328-002
**问题**: 单元测试随机失败
**优先级**: P1
**状态**: 🛠️ 修复中

## 问题现象
- test_user_creation 测试用例随机失败
- 错误信息：AssertionError: 预期 user.is_active=True，实际为 False
- 失败率约 10%，无法稳定复现

## 排查过程
1. **分析失败日志**: 失败时 user 对象已创建但 is_active=False
2. **检查测试代码**: 发现测试依赖全局状态（时间戳）
3. **检查并发**: CI 环境并行执行测试，可能存在竞争
4. **添加日志**: 发现测试清理函数会修改其他测试的数据

## 根因分析
**根本原因**: 测试污染
- tearDown 清理数据时误删其他测试的数据
- 测试之间没有完全隔离

## 修复方案
```python
# 修复前：全局清理
def tearDown(self):
    User.query.delete()  # 删除所有用户

# 修复后：只删除自己的数据
def tearDown(self):
    if hasattr(self, 'created_user_id'):
        User.query.filter_by(id=self.created_user_id).delete()
```

## 验证结果
- [x] 本地运行 100 次通过
- [ ] CI 运行 50 次通过（进行中）

**需要同步的部门**:
@QA @Engineering
```

**输入**:

**环境**: 生产环境

**输出**:
```markdown
📋 问题排查报告

**问题 ID**: BUG-20260328-003
**问题**: 偶尔页面加载失败
**优先级**: P2
**状态**: 🔍 分析中

## 问题现象
- 用户反馈偶尔（约 1% 请求）页面加载超时
- 错误信息：Gateway Timeout
- 无法稳定复现

## 已尝试排查
1. **查看应用日志**: 无明显错误
2. **检查服务器资源**: CPU/内存正常
3. **分析慢请求日志**: 失败请求响应时间>30 秒
4. **检查第三方服务**: 外部 API 响应时间正常

## 假设和验证计划
| 假设 | 验证方法 | 状态 |
|------|----------|------|
| 网络波动 | 分析网络监控数据 | 待验证 |
| 数据库锁 | 检查锁等待日志 | 待验证 |
| GC 停顿 | 分析 JVM GC 日志 | 待验证 |
| 连接池耗尽 | 检查连接池监控 | 待验证 |

## 下一步行动
1. 添加详细日志记录（请求 ID、时间戳）
2. 部署到预发环境，进行压力测试
3. 监控 24 小时，收集失败样本

## 临时措施
- 增加请求超时时间（15s → 30s）
- 添加前端重试机制

**需要同步的部门**:
@CTO @QA
